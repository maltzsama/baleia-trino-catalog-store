### N5 â€” `@baleia-secret[...]` placeholder syntax â€” collision found and fixed

Verified in 482 source: the DDL path is

```
CreateCatalogTask.execute
    â†’ MetadataManager.createCatalog       (MetadataManager.java:849)
        â†’ CatalogManager.createCatalog    (CatalogManager.java:55)
            â†’ CoordinatorDynamicCatalogManager.createCatalog   (CoordinatorDynamicCatalogManager.java:257)
                â†’ catalogStore.createCatalogProperties(...)    (line 273)
                â†’ catalogStore.addOrReplaceCatalog(...)        (line 280, after the in-memory register)
```

`CreateCatalogTask` evaluates each property with `evaluateProperty(...)` (a SQL
expression evaluator) and passes the resulting string map straight through to
`MetadataManager.createCatalog`. **No `secretsResolver.getResolvedConfiguration`**
appears on this path on the *server* side.
So a placeholder travels in verbatim â€” only our own `SecretResolver`
interprets it in `BaleiaCatalogStore.createCatalogProperties`.

**Collision found during T8: the Trino CLI itself eats `${...}` before
the SQL reaches the server.** The CLI picocli layer treats `${baleia-secret:...}`
as a variable expansion and strips it. Running

```
trino --output-format TSV --execute "CREATE CATALOG seg USING tpch WITH (\"k\" = '\${baleia-secret:catalog:key}')"
```

sent `"secret:catalog:key"` (the default-value portion of a picocli variable)
to `TpchConfig.setSplitsPerNode(int)`, which rejected it as not-an-int. The
server-side N5 check was correct â€” the damage is on the *client* side and was
not caught by the grego-based review because the review was syntactic, not
empirical.

The placeholder syntax was therefore changed
from `${baleia-secret:<catalog>:<key>}` to
`@baleia-secret[<catalog>:<key>]`. The `@square-brackets` form is outside every
standard expansion syntax (bash, picocli, Airlift, Spring), so it survives any
client in the ecosystem. The Java side owns the regex in
`SecretResolver.PLACEHOLDER`; the Go side will own the DDL generator that emits
the placeholder.

â†’ On Trino upgrade, re-grep the four classes above for any new
`secretsResolver` / `getResolvedConfiguration` reference, and *also* re-run
`acceptance.sh T8` against the CLI to confirm the new syntax still passes the CLI
without mangling. If a future Trino CLI introduces its own `@baleia-secret[...]`
expansion (vanishingly unlikely), pick another non-`${...}` shape and update
both the Java regex and the Go DDL generator.

## `FileCatalogStore` (reference: the built-in implementation we copy the skeleton from)

- `computeCatalogVersion` is **SHA-256** with:
  ```java
  hasher.putUnencodedChars("catalog-hash");          // fixed prefix
  hashLengthPrefixedString(catalogName.toString());
  hashLengthPrefixedString(connectorName.toString());
  hasher.putInt(properties.size());
  ImmutableSortedMap.copyOf(properties).forEach((k, v) -> {
      hashLengthPrefixedString(k);
      hashLengthPrefixedString(v);
  });
  return new CatalogVersion(hasher.hash().toString());

  private static void hashLengthPrefixedString(Hasher hasher, String value) {
      hasher.putInt(value.length());            // little-endian (Guava)
      hasher.putUnencodedChars(value);          // UTF-16LE (Guava)
  }
  ```
- This is the algorithm the Go implementation must reproduce **byte-for-byte**. Warning:
  little-endian + UTF-16LE confirmed (inherited from Guava `Hasher`).
- `createCatalogProperties(catalogName, connectorName, properties)` builds:
  ```java
  new CatalogProperties(catalogName, computeCatalogVersion(catalogName, connectorName, properties), connectorName, ImmutableMap.copyOf(properties))
  ```
  â†’ No `CatalogHandle` involved. The version is the truth.
- In `addOrReplaceCatalog`: `catalogProperties.name()`, `connectorName()`, `version()`, `properties()`.
- In loadProperties: removes `connector.name` from the loaded properties (here the DB already has it separate).
- On persistence error, throws `TrinoException(CATALOG_STORE_ERROR, "Could not store catalog properties")`.

## Operational summary for the plugin

- Implement `CatalogStore` + `CatalogStoreFactory` + `Plugin.getCatalogStoreFactories()`.
- `createCatalogProperties(CatalogName, ConnectorName, Map)` â†’ resolves secrets, computes SHA-256
  version, returns `new CatalogProperties(name, version, connector, ImmutableMap.copyOf(resolved))`.
- `addOrReplaceCatalog(CatalogProperties)` â†’ UPSERTs the row with `version().toString()`.
- `removeCatalog(CatalogName)` â†’ soft delete.
- `getCatalogs()` â†’ SELECT enabled rows; `StoredCatalog.loadProperties()` resolves secrets and
  returns `CatalogProperties` consistent on `name()`.
- On persistence error: `throw new TrinoException(CATALOG_STORE_ERROR, "...")` (do not swallow).
